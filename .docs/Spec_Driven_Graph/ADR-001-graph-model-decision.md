# ADR-001: النموذج البياني المختلط لوثائق Smart Task

- **الحالة**: Accepted
- **التاريخ**: 2026-06-02
- **التصنيف**: architecture-change
- **النسخة**: 1

## السياق

كان مشروع Smart Task يستخدم في البداية **ملفات Markdown غير منظمة** كمواصفات للتطوير. مع توسع المشروع، ظهرت المشاكل التالية:

1. **عدم وجود تكامل مرجعي**: لا يمكن تتبع أي مواصفة تشير إلى أي وثيقة أخرى.
2. **صعوبة الاستعلام**: لا يمكن السؤال "ما هي كل المهام المشتقة من هذه المواصفة؟" دون بحث يدوي.
3. **ازدواجية المعلومات**: المواصفات تتكرر عبر ملفات متعددة بدون آلية مزامنة.
4. **غياب التتبع**: لا يمكن تتبع دورة حياة الوثيقة من الإنشاء إلى الموافقة إلى التنفيذ.

## القرار

اعتماد **نموذج بياني مختلط** يتكون من جزئين متكاملين:

### الجزء الأول: جداول منفصلة لكل فئة وثيقة مع FK مباشر

كل فئة من الوثائق (PRD, Architecture, Spec, Spec Topic, Interface, ADR, Plan, Execution Plan, Audit Report) تحصل على **جدول SQLite خاص بها** مع:

- حقل `id` بنظام البادئات (PRD-001, ARCH-001, إلخ)
- حقل FK يشير إلى **الوالد المباشر** في السلسلة لضمان التكامل المرجعي
- حقول metadata شائعة: status, version, tags, timestamps
- حقول خاصة بكل فئة حسب طبيعتها

### الجزء الثاني: جدول حواف عام (doc_edges)

جدول تجاور (adjacency list) يدعم أي علاقة بين أي وثيقتين:

```sql
(source_type, source_id, target_type, target_id, edge_type)
```

مع أنواع علاقات محددة مسبقًا: `parent`, `child`, `references`, `informs`, `implements`, `supersedes`, `sibling`, `decomposes`.

### السلسلة الهرمية الأساسية

```
PRD
 └── Architecture (FK → PRD)
      └── Spec (FK → Architecture)
           └── Spec Topic (FK → Spec)
                ├── Interface (FK → Spec Topic)
                └── Task (FK → Spec Topic)
                     └── Wave
                          └── Execution Plan (FK → Wave)
                               └── Audit Report (FK → Execution Plan)
```

### استثناء ADR

ADR لا يحتوي على FK مباشر لأنه يمكن أن يشير إلى أي نوع من الوثائق. بدلاً من ذلك، يستخدم حقل `target_id` نصي ويرتبط عبر `doc_edges`.

## النتائج

### الإيجابية (+)

1. **تتبع كامل**: كل وثيقة يمكن تتبعها إلى جذر السلسلة (PRD) باستخدام JOINs متسلسلة.
2. **إمكانية الاستعلام**: يمكن استخدام SQL للاستعلام عن العلاقات — مثلاً "كل المهام المرتبطة بموضوع معين".
3. **مرونة العلاقات**: doc_edges تسمح بأي علاقة لا تدعمها المفاتيح الخارجية وحدها (علاقات أفقية، cross-references).
4. **قابلية التوسع**: يمكن إضافة أنواع جديدة من العلاقات أو فئات جديدة من الوثائق دون تغيير المخطط الأساسي.
5. **التكامل مع SQLite**: النموذج يعمل بكفاءة ضمن قيود SQLite (لا يدعم JSONB أو graph extensions).

### السلبية (-)

1. **لا يوجد FK enforcement في doc_edges**: العلاقات في جدول الحواف لا تفرض تكاملاً مرجعياً على مستوى قاعدة البيانات — يجب التعامل مع هذا في طبقة التطبيق.
2. **تعقيد إضافي**: الحاجة إلى صيانة جدولين (FK + edges) يزيد من تعقيد طبقة الوصول إلى البيانات.
3. **تكرار العلاقات الأساسية**: العلاقة بين الوالد والابن موجودة في كل من FK المباشر و doc_edges — يجب ضمان التناسق بينهما.
4. **حجم البيانات**: جدول doc_edges يمكن أن ينمو بسرعة مع زيادة عدد الوثائق والعلاقات.

## البدائل التي تم النظر فيها

### البديل 1: جدول موحد للوثائق (Single Unified Table)

```
documents(id, type, title, content, parent_id, ...)
```

- **المزايا**: بسيط، جدول واحد لكل شيء.
- **العيوب**: عمود `content` يصبح غير متجانس (JSON mixed مع نصوص مختلفة)، لا يمكن فرض schema خاص بكل نوع، أداء ضعيف مع كثرة الصفوف.
- **الرفض**: بسبب فقدان سلامة النوع (type safety) وصعوبة إدارة الحقول الخاصة بكل فئة.

### البديل 2: FK-only (هرمي صرف)

```sql
specs(arch_id FK → architecture)
architecture(prd_id FK → prd)
-- بدون doc_edges
```

- **المزايا**: بسيط، تكامل مرجعي كامل.
- **العيوب**: لا يدعم العلاقات الأفقية (sibling)، لا يدعم cross-references (ADR → أي وثيقة)، لا يدعم العلاقات المتعددة.
- **الرفض**: بسبب عدم كفايته لتمثيل العلاقات المعقدة في نظام حقيقي.

### البديل 3: Edge-only (بياني خالص)

```sql
nodes(id, type, data)
edges(source_id, target_id, type)
```

- **المزايا**: مرونة كاملة، أي علاقة ممكنة.
- **العيوب**: فقدان كامل للتكامل المرجعي، استعلامات معقدة (CTE متكررة)، أداء أقل، صعوبة في إنشاء تقارير SQL بسيطة.
- **الرفض**: بسبب تعقيد الاستعلامات اليومية وفقدان التكامل المرجعي.

## القرار النهائي

تم اعتماد **النموذج المختلط** لأنه يوفر أفضل توازن بين:
- **التكامل المرجعي**: FK المباشر يضمن صحة البيانات للعلاقات الأساسية.
- **المرونة**: doc_edges يسمح بأي علاقة إضافية.
- **الأداء**: استعلامات السلسلة الأساسية سريعة باستخدام JOINs مباشرة.
- **قابلية الفهم**: المخطط واضح ومنطقي لأعضاء الفريق الجدد.
