# دستور المشروع (Constitution)

## تعريف

`constitution` هو **الدستور** — القواعد الثابتة والاتفاقيات المعمارية والبرمجية للمشروع. يضمن الاتساق عبر الفرق والأفراد من خلال توثيق المبادئ التي لا يجوز خرقها إلا بعد تعديل الدستور نفسه.

## موقعه في النظام

- **فئة**: cross-cutting document class
- **parent FK**: لا يوجد — ليس له أب في السلسلة الرأسية
- **عدد المستندات**: يمكن أن يكون مستندًا واحدًا أو عدة إصدارات (versions)
- **is_mandatory**: 1 (إلزامي — كل مشروع يجب أن يملك دستورًا)

الـ Constitution مستقل عن السلسلة الرأسية PRD → ARCH → SPEC → ... لأنه يطبق على **كل** الوثائق و**كل** الكود، وليس على مرحلة معينة.

## الفرق بينه وبين ADR

| البعد | Constitution | ADR |
|-------|-------------|-----|
| **الطبيعة** | قواعد ثابتة مستمرة | قرارات معماريّة في نقطة زمنية |
| **التغيير** | نادر — يتطلب تصديقًا | يمكن أن يتغير بقرار جديد |
| **النطاق** | يغطي المشروع بأكمله | قد يغطي جزءًا معينًا |
| **مثال** | "جميع الاستثناءات يجب أن تكون مخصصة" | "اخترنا PostgreSQL لدعم JSONB" |
| **العمر** | يعيش طول عمر المشروع | قد يُهمَل أو يُستبدَل بعد التنفيذ |

## الفرق بينه وبين Cross-Cutting Spec

| البعد | Constitution | Cross-Cutting Spec |
|-------|-------------|-------------------|
| **المحتوى** | قواعد والتزامات | مواصفات عرضية (مثل performance, security) |
| **السلوك** | إلزامي — violations تحتاج تعديل الدستور | توجيهي — specs يمكن تجاهلها بقرار |
| **العلاقة** | الدستور يحكم كيف تُكتب الـ specs | الـ specs تحدد ماذا يجب أن يفعل النظام |
| **المخاطب** | المطورون والمعماريون ومديرو المشروع | المعماريون وفريق الجودة |

## DDL

```sql
CREATE TABLE constitutions (
    id                  TEXT PRIMARY KEY,       -- CONST-001
    title               TEXT NOT NULL,
    status              TEXT NOT NULL CHECK (status IN ('draft', 'ratified', 'amended')),
    articles            TEXT NOT NULL,           -- JSON: array of articles (rules)
    tags                TEXT NOT NULL DEFAULT '[]',
    version             TEXT NOT NULL DEFAULT '1.0',
    created_at          TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at          TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## شرح الحقول

| الحقل | النوع | إلزامي | مثال JSON | الغرض |
|-------|-------|--------|-----------|-------|
| `id` | TEXT | نعم | `"CONST-001"` | معرف فريد — البادئة CONST متبوعة برقم تسلسلي |
| `title` | TEXT | نعم | `"دستور مشروع Smart Task"` | عنوان وصفي للدستور |
| `status` | TEXT | نعم | `"ratified"` | حالة الدستور: draft (مسودة) → ratified (مُصدّق) → amended (مُعدّل) |
| `articles` | TEXT | نعم | `[{"article_id":"ART-001","category":"coding_standard","title":"..."}]` | مصفوفة المواد (القواعد) — الحقل الأساسي الذي يحتوي على جميع بنود الدستور |
| `tags` | TEXT | نعم | `["coding-standards", "security"]` | وسوم لتصنيف وتحليل المواد |
| `version` | TEXT | نعم | `"1.0"` | إصدار الدستور — يزداد مع كل تعديل |
| `created_at` | TEXT | نعم | `"2025-01-01T00:00:00Z"` | طابع زمني للإنشاء |
| `updated_at` | TEXT | نعم | `"2025-06-01T00:00:00Z"` | طابع زمني لآخر تحديث |

## هيكل عنصر articles (كل Article/مادة)

كل عنصر في مصفوفة `articles` هو كائن JSON يمثّل قاعدة واحدة:

| الحقل | النوع | إلزامي | مثال | الغرض |
|-------|-------|--------|------|-------|
| `article_id` | string | نعم | `"ART-001"` | معرف فريد للمادة |
| `category` | string | نعم | `"coding_standard"` | فئة القاعدة — تحدد طبيعة الالتزام |
| `title` | string | نعم | `"تسمية المتغيرات"` | عنوان مختصر للقاعدة |
| `statement` | string | نعم | `"جميع أسماء المتغيرات بـ snake_case"` | نص القاعدة كاملاً |
| `rationale` | string | نعم | `"ضمان قراءة موحّدة للكود عبر الفريق"` | السبب المنطقي وراء القاعدة |
| `enforcement` | string | نعم | `"automated"` | طريقة الإنforcement: automated (آلي)، manual (يدوي)، advisory (استرشادي) |
| `applies_to` | array | لا | `["python", "sql"]` | أنظمة/لغات تنطبق عليها القاعدة. `["all"]` للقواعد العامة |
| `effective_date` | string | لا | `"2025-01-01"` | تاريخ بدء سريان القاعدة |
| `supersedes` | string | لا | `"ART-000"` | article ID قديم تم استبداله بهذه القاعدة (إن وجد) |

## تصنيف articles — شرح الفئات مع أمثلة

### `coding_standard`
معايير كتابة الكود — التنسيق، التسمية، الهيكل.

| مثال | الشرح |
|------|-------|
| `"جميع أسماء المتغيرات بـ snake_case"` | يضمن اتساق أسماء المتغيرات عبر قاعدة الكود |
| `"أقصى طول للدالة 50 سطرًا"` | يحافظ على قابلية القراءة |
| `"نوع الإرجاع مصري (type hint) لكل دالة"` | يضمن type safety |
| `"الاستثناءات يجب أن تكون كلاسات مخصصة — لا نرمي str"` | يحسن error handling |
| `"جميع الـ imports مطلقة (absolute) وليست نسبية"` | يمنع ambiguity |

### `architectural_principle`
مبادئ معمارية أساسية تحكم تصميم النظام.

| مثال | الشرح |
|------|-------|
| `"الاعتماد من الخارج إلى الداخل فقط (Dependency Rule)"` | الـ layers الداخلية لا تعرف شيئًا عن الخارجية |
| `"الـ Domain لا يعرف شيئًا عن الـ Infrastructure"` | تطبيق مبدأ Clean Architecture |
| `"كل خدمة تملك قاعدة بيانات مستقلة (Database per Service)"` | يمنع coupling بين الخدمات |
| `"التواصل بين الـ modules يتم عبر واجهات فقط (وليس كائنات مباشرة)"` | يفرض loose coupling |

### `naming_convention`
اصطلاحات التسمية للوثائق والكيانات والملفات.

| مثال | الشرح |
|------|-------|
| `"بادئات الوثائق: PRD-، ARCH-، SPEC-، TEST-، CONST-"` | يسهل التعرف على نوع الوثيقة من المعرف |
| `"أسماء ملفات الوثائق بـ snake_case.md"` | تسمية موحدة للملفات |
| `"أسماء الجداول بصيغة الجمع (users, orders, ...)"` | يضمن اتساق قاعدة البيانات |
| `"أسماء الـ branches: feature/، fix/، chore/"` | اتفاقية git |

### `team_convention`
اتفاقيات فريق العمل.

| مثال | الشرح |
|------|-------|
| `"مراجعة الكود إلزامية لكل PR"` | ضمان الجودة |
| `"كل commit يجب أن يشير إلى Task ID"` | إمكانية التتبع |
| `"الـ PR لا يقل عن مراجعَين اثنين"` | توزيع المعرفة |
| `"الاجتماعات اليومية قبل الساعة 10 صباحًا"` | تنظيم العمل |

### `process_rule`
قواعد إجرائية — كيف يُدار العمل.

| مثال | الشرح |
|------|-------|
| `"لا يُعتبر Task منجزًا بدون Test من فئة capability_audit"` | ضمان اختبار كل مهمة |
| `"لا يُدمج PR دون اجتياز CI/CD بالكامل"` | حماية الـ main branch |
| `"أي فجوة critical في الاختبار تُنتج ADR إجباريًا"` | ربط الاختبارات بالقرارات المعمارية |
| `"الـ Audit Report يُنفذ عند اكتمال كل Wave"` | دورة حكم دورية |

### `security_policy`
سياسات أمنية ثابتة.

| مثال | الشرح |
|------|-------|
| `"لا تُخزن المفاتيح الخاصة في الكود"` | منع تسرب credentials |
| `"كل API يتطلب مصادقة (Authentication)"` | حماية endpoints |
| `"تعقيم المدخلات (Input Sanitization) إلزامي لكل واجهة"` | منع injection attacks |
| `"الـ secrets تُدار عبر vault (وليس env files)"` | إدارة آمنة للأسرار |

## دورة حياة الربط

```
1. Constitution يُصدّق (ratified) ← يصبح المرجع الوحيد للقواعد
2. أي وثيقة تستند إلى قاعدة معينة ← تُنشئ edge references → Constitution
3. الـ Execution Plan يستند إلى articles ← references → مادة معينة
4. الـ Audit Report يتحقق من الالتزام ← يفحص كل article مقابل السلوك الفعلي
5. إن وُجد violation ← يُسجل في audit report ويُنتج ADR أو Task للمعالجة
6. عند الحاجة لتعديل قاعدة ← يُنشأ إصدار جديد من Constitution (status = amended)
7. الإصدار القديم يبقى للتوثيق التاريخي
```

### الحواف المسموحة مع Constitution

| المصدر | نوع الحافة | الهدف | المعنى |
|--------|-----------|-------|--------|
| `CONST-001` | `references` | `ART-001` | الإشارة إلى مادة معينة |
| `EP-001` | `references` | `CONST-001` | خطة التنفيذ تستند إلى مواد الدستور |
| `AR-001` | `informed_by` | `CONST-001` | تقرير التدقيق يسترشد بمواد الدستور |
| `AR-001` | `references` | `ART-001` | تدقيق مادة معينة |

## دورة حالة Constitution

```
draft → ratified → amended
```

- `draft`: مسودة قيد الصياغة — المواد تُكتب وتُنقّح
- `ratified`: تم التصديق — الدستور نافذ المفعول
- `amended`: تم التعديل — إصدار جديد يحل محل القديم

## مصدر الحقيقة

الـ `constitutions` (جدول) هو السجل الرسمي للدستور. أي قاعدة جديدة أو تعديل على قاعدة موجودة يجب أن يُسجل أولاً في جدول `constitutions` قبل تطبيقه. لا يُعترف بأي قاعدة غير مسجلة في الدستور.

الـ `doc_edges` هو مصدر الحقيقة للربط — أي وثيقة تستند إلى الدستور تُسجل عبر edge من نوع `references`.
